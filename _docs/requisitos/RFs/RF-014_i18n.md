## 📚 Conceptos Clave (para aprendices)

| Concepto          | Definición                                                                                |
| ----------------- | ----------------------------------------------------------------------------------------- |
| **i18n**          | Internacionalización — diseño del sistema para soportar múltiples idiomas sin refactoring |
| **l10n**          | Localización — proceso de añadir traducciones y formatos regionales a cada idioma         |
| **locale**        | Identificador de idioma/región. Ejemplos: `es`, `en`, `es-CO`, `en-US`                    |
| **namespace**     | Agrupación lógica de claves de traducción (usamos un único namespace `translation`)       |
| **interpolación** | Insertar variables dinámicas dentro de un texto traducido: `"Hola, {{name}}"`             |
| **fallback**      | Idioma al que se recurre si no se encuentra una clave en el idioma activo                 |

---

## Descripción General

El sistema NN Auth debe soportar dos idiomas: **español (`es`)** e **inglés (`en`)**.
Español es el idioma por defecto (proyecto SENA Colombia).
Los textos de la interfaz deben cambiar de forma dinámica sin recargar la página.
La preferencia se guarda en `localStorage` (cliente) y en la columna `locale` de la BD (servidor).

## Requisitos Funcionales

### RF-014.1 — Idiomas Soportados

El sistema debe soportar exactamente dos locales:

| Locale | Idioma  | Región           |
| ------ | ------- | ---------------- |
| `es`   | Español | Colombia (LATAM) |
| `en`   | English | General          |

**Archivo de traducción:** `src/locales/{locale}/translation.json`.

### RF-014.2 — Detección Automática del Idioma

Al cargar por primera vez (sin preferencia guardada), el frontend debe:

1. Leer `navigator.language` del navegador (ej: `"es-CO"`, `"en-US"`).
2. Extraer la parte base del locale (`"es"`, `"en"`).
3. Si el base locale está en la lista de soportados → usarlo.
4. Si no → usar `"es"` como fallback.

Herramienta: `i18next-browser-languagedetector`.

### RF-014.3 — Almacenamiento en localStorage

Toda preferencia de idioma (detectada o elegida manualmente) debe guardarse en
`localStorage` bajo la clave `i18nextLng`.

Al recargar la página, `i18next-browser-languagedetector` lee esa clave automáticamente.

### RF-014.4 — Columna `locale` en tabla `users`

Agregar columna en la tabla `users`:

```sql
ALTER TABLE users ADD COLUMN locale VARCHAR(10) NOT NULL DEFAULT 'es';
```

| Propiedad | Valor                                               |
| --------- | --------------------------------------------------- |
| Nombre    | `locale`                                            |
| Tipo      | `VARCHAR(10)`                                       |
| Default   | `'es'`                                              |
| Nullable  | `NOT NULL`                                          |
| Valores   | Solo `'es'` o `'en'` (validado en capa de servicio) |

### RF-014.5 — Endpoint PATCH /api/v1/users/me/locale

| Campo    | Valor                                       |
| -------- | ------------------------------------------- |
| Método   | `PATCH`                                     |
| Ruta     | `/api/v1/users/me/locale`                   |
| Auth     | Requerida (`Authorization: Bearer <token>`) |
| Body     | `{"locale": "es"}` o `{"locale": "en"}`     |
| Response | `UserResponse` actualizado (200 OK)         |
| Errores  | 401 (no autenticado), 422 (locale inválido) |

**Schema de request:**

```python
class UpdateLocaleRequest(BaseModel):
    locale: str

    @field_validator("locale")
    @classmethod
    def validate_locale(cls, v: str) -> str:
        if v not in ("es", "en"):
            raise ValueError("Locale no soportado. Use 'es' o 'en'.")
        return v
```

### RF-014.6 — Sincronización Frontend → Backend

**Al cambiar idioma (usuario autenticado):**

```
Usuario hace clic en LanguageSwitcher
  → i18n.changeLanguage(locale)  ← cambia la UI inmediatamente
  → localStorage.setItem("i18nextLng", locale)  ← persiste en cliente
  → PATCH /api/v1/users/me/locale { locale }  ← sincroniza en BD
```

**Al iniciar sesión:**

```
Login exitoso → Response incluye user.locale
  → i18n.changeLanguage(user.locale)  ← aplica preferencia guardada
  → localStorage.setItem("i18nextLng", user.locale)
```

### RF-014.7 — Componente LanguageSwitcher

El componente `LanguageSwitcher` debe:

- Mostrarse en la `Navbar` junto al `ThemeToggle`.
- Mostrar las dos opciones de idioma: "Español" / "English".
- Indicar visualmente el idioma activo (ej: texto resaltado o badge).
- Al seleccionar, llamar a `i18n.changeLanguage()` y sincronizar con el backend si hay sesión.
- Ser accesible: `aria-label="Selector de idioma"` y opciones con `lang` attribute.

### RF-014.8 — Catálogo de Traducciones

Todos los textos visibles de la interfaz deben estar en los archivos de traducción.
**Ningún texto debe estar hardcodeado** en los componentes.

**Estructura jerárquica de claves:**

```
nav.*             → barra de navegación
common.*          → textos reutilizables (loading, email, password, etc.)
auth.login.*      → página de inicio de sesión
auth.register.*   → página de registro
auth.changePassword.* → página de cambio de contraseña
auth.forgotPassword.* → página de recuperación de contraseña
auth.resetPassword.*  → página de restablecimiento de contraseña
dashboard.*       → panel de control
language.*        → textos del selector de idioma
```

## Reglas de Negocio

| ID     | Regla                                                                                                                     |
| ------ | ------------------------------------------------------------------------------------------------------------------------- |
| RN-073 | El idioma por defecto del sistema es `es` (español).                                                                      |
| RN-074 | El sistema soporta exactamente dos locales: `es` y `en`.                                                                  |
| RN-075 | La preferencia de idioma se guarda siempre en `localStorage` (clave `i18nextLng`).                                        |
| RN-076 | Si el usuario está autenticado, el cambio de idioma se sincroniza con el backend via `PATCH /api/v1/users/me/locale`.     |
| RN-077 | Al restaurar sesión (token válido), se aplica el `locale` del usuario desde la BD.                                        |
| RN-078 | Los textos del `LanguageSwitcher` se muestran en el idioma propio de cada opción, no en el idioma activo.                 |
| RN-079 | El cambio de idioma es inmediato — sin recarga de página.                                                                 |
| RN-080 | El backend rechaza valores de `locale` distintos a `es` y `en` con HTTP 422.                                              |
| RN-081 | Nuevos usuarios se registran con `locale='es'` por defecto.                                                               |
| RN-082 | El `LanguageSwitcher` debe ser accesible: usa `aria-label` y atributo `lang` en opciones.                                 |
| RN-083 | Las claves de traducción se escriben en inglés, en `camelCase`.                                                           |
| RN-084 | Los valores de traducción se escriben en el idioma correspondiente al archivo.                                            |
| RN-085 | Las variables dinámicas en traducciones usan la sintaxis `{{variable}}` de i18next.                                       |
| RN-086 | Se usa un único namespace `translation` para todos los archivos (simplicidad pedagógica).                                 |
| RN-087 | El archivo de configuración de i18n (`src/i18n.ts`) importa los recursos de forma estática (no lazy-load).                |
| RN-088 | En tests, se usa un mock sincrono de i18next que retorna las traducciones en español.                                     |
| RN-089 | No se usa el locale con región (`es-CO`, `en-US`) — solo el base (`es`, `en`).                                            |
| RN-090 | Si falla la llamada PATCH al backend (error de red o token expirado), el cambio de idioma en el frontend igual se aplica. |

## Flujo completo de i18n

```
┌─────────────────────────────────────────────────────────────────┐
│                    FLUJO DE INTERNACIONALIZACIÓN                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│  PRIMERA VISITA (sin preferencia)                                 │
│  ─────────────────────────────                                    │
│  App carga → i18next detecta navigator.language                   │
│           → Si es "es" o "en" → usa ese locale                    │
│           → Si no es soportado → usa "es" (fallback)              │
│           → Guarda en localStorage["i18nextLng"]                  │
│                                                                   │
│  CAMBIO MANUAL (usuario hace clic en LanguageSwitcher)            │
│  ──────────────────────────────────────────────────               │
│  Clic en "English"                                                │
│    → i18n.changeLanguage("en")  ← UI cambia inmediatamente        │
│    → localStorage["i18nextLng"] = "en"                            │
│    → Si autenticado: PATCH /api/v1/users/me/locale {"locale":"en"} │
│                      → BD: users.locale = "en"                    │
│                                                                   │
│  LOGIN (usuario autenticado)                                      │
│  ──────────────────────────                                       │
│  POST /auth/login → Response: {..., locale: "en"}                 │
│    → i18n.changeLanguage("en")  ← aplicar preferencia del usuario │
│    → localStorage["i18nextLng"] = "en"                            │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```
